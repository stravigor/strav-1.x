# Inbound mail

`@strav/mail` ships parsers for two inbound-webhook providers: Postmark and Mailgun. Both parse to the same shape — `ParsedInboundMail` — so application code stays provider-agnostic. Other parsers (SendGrid, SES, Resend) can be added later without touching call sites.

## The normalised shape

```ts
interface ParsedInboundMail {
  from: ParsedInboundAddress
  to: ParsedInboundAddress[]
  cc: ParsedInboundAddress[]
  bcc: ParsedInboundAddress[]
  replyTo?: ParsedInboundAddress
  subject: string
  text?: string
  html?: string
  date?: Date
  headers: Record<string, string>      // lowercased; last value wins on dupes
  attachments: ParsedInboundAttachment[]
  messageId?: string                   // RFC 5322 Message-ID, angle brackets stripped
  inReplyTo?: string
  references: string[]
  isAutoGenerated: boolean             // honor before any auto-reply
  providerMessageId?: string           // e.g. Postmark MessageID
}
```

Provider-specific quirks are flattened here: Postmark's `MailboxHash` is dropped (read it off `to[0].address` if you encode IDs into the local-part), Mailgun's `message-headers` JSON is parsed into the same `headers` map every other provider produces, attachment bytes land as Node `Buffer` regardless of how the provider delivered them.

## Wiring the webhook route

Both parsers take `{ body, headers }` and return `ParsedInboundMail`. Pass the raw request body — not the parsed one — because signature verification (Mailgun) needs the exact bytes the provider signed.

```ts
// app/Controllers/inbound_mail_controller.ts
import { inject } from '@strav/kernel'
import { AuthError } from '@strav/kernel'
import { MailgunInboundParser, MailInboundError } from '@strav/mail'
import type { ParsedInboundMail } from '@strav/mail'
import { SupportTicketService } from '../Services/support_ticket_service.ts'

@inject()
export class InboundMailController {
  private readonly parser = new MailgunInboundParser({
    webhookSigningKey: process.env.MAILGUN_SIGNING_KEY ?? '',
  })

  constructor(private readonly tickets: SupportTicketService) {}

  async receive(req: Request): Promise<Response> {
    let mail: ParsedInboundMail
    try {
      mail = await this.parser.parse({
        body: Buffer.from(await req.arrayBuffer()),
        headers: lowercaseHeaders(req.headers),
      })
    } catch (err) {
      if (err instanceof AuthError) return new Response(null, { status: 401 })
      if (err instanceof MailInboundError) return new Response(null, { status: 400 })
      throw err
    }

    if (mail.isAutoGenerated) return new Response(null, { status: 200 })

    await this.tickets.fromInbound(mail)
    return new Response(null, { status: 200 })
  }
}

function lowercaseHeaders(h: Headers): Record<string, string | undefined> {
  const out: Record<string, string | undefined> = {}
  h.forEach((value, key) => {
    out[key.toLowerCase()] = value
  })
  return out
}
```

Three details worth knowing:

- **Headers must be lowercased.** Both parsers index headers by lowercase key (`content-type`, `auto-submitted`). If you forward `req.headers` directly without normalising, Mailgun's signature check fails because the parser can't find `content-type`.
- **Return 200 on auto-generated.** Returning non-2xx tells the provider to retry — and an auto-reply to a vacation responder is exactly the loop you don't want. Acknowledge the delivery, then drop the message.
- **Return 401 on `AuthError`.** Mailgun retries on 5xx and timeouts; a clean 401 stops the redelivery and is the correct shape if the signing key has rotated and you need to re-deploy with the new value.

## Postmark — no signature, HTTP-layer auth

Postmark does NOT sign inbound webhooks. Protect the route at the HTTP layer before invoking the parser:

```ts
const parser = new PostmarkInboundParser()
```

Two common patterns:

1. **Basic auth in the webhook URL.** In the Postmark console, set the inbound webhook URL to `https://user:pass@inbox.acme.com/webhooks/postmark`. Your HTTP middleware verifies the `Authorization: Basic ...` header. The credentials never appear in your application code.
2. **IP allow-listing.** Postmark publishes its outbound IP ranges; your edge (nginx / Cloudflare / etc.) rejects anything outside that range. Cheaper than verifying on every request but tied to Postmark's network operations.

Either way, the parser itself trusts the body — `parse()` will happily parse a forged JSON payload if you don't gate the route. Treat `PostmarkInboundParser` as a pure decoder, not an authentication step.

The only error `PostmarkInboundParser` throws is `MailInboundError` on invalid JSON. The corresponding HTTP response is 400.

## Mailgun — HMAC-SHA256 + replay window

Mailgun signs every webhook with HMAC-SHA256 over `timestamp + token`, keyed by the dashboard's webhook signing key. The parser verifies the signature and rejects timestamps outside a configurable replay window:

```ts
const parser = new MailgunInboundParser({
  webhookSigningKey: process.env.MAILGUN_SIGNING_KEY ?? '',
  maxAgeSeconds: 300,    // default 300 (5 minutes) — tighten if your clocks are reliable
})
```

The signing key is distinct from your Mailgun sending API key — rotate them independently. The parser uses Node's `crypto.timingSafeEqual` so signature comparison is constant-time.

Errors:

- `AuthError` — missing signature/token/timestamp fields, signature mismatch, or timestamp outside the replay window. Respond 401.
- `MailInboundError` — non-multipart body, or unparseable multipart. Respond 400.
- `ConfigError` (at construction) — empty `webhookSigningKey`. Surfaces at boot, not at first webhook.

Mailgun's webhook delivery retries failed deliveries with exponential backoff. A returned 200 acknowledges receipt; a 401 stops the retry; a 5xx (or no response) triggers redelivery for several hours.

## The mail-loop guard

`isAutoGeneratedMessage(headers)` honours three universal signals:

- `Auto-Submitted` (RFC 3834) — anything other than `no` means auto-generated.
- `Precedence: bulk | junk | list` — list mail, vacation responders, marketing blasts.
- `X-Auto-Response-Suppress` (any value) — Microsoft / Outlook convention.

Always check this before any auto-reply path:

```ts
if (mail.isAutoGenerated) return    // do NOT respond
```

Without it, an auto-reply to a vacation responder triggers their auto-reply to your auto-reply, ad infinitum until one provider gives up. The cost of skipping the check is a runaway mail loop with both providers; the cost of honouring it is dropping a small number of legitimate-but-unusual messages. Honour it.

The function is exported separately so you can apply it to mail that came from another path — e.g. an IMAP-pulled message — without going through one of the shipped webhook parsers.

## Threading

Three fields cover the standard RFC 5322 threading triple:

- `messageId` — the inbound message's own Message-ID, angle brackets stripped.
- `inReplyTo` — the message this one is replying to.
- `references` — the full ancestry chain, oldest first; brackets stripped on each entry.

Common pattern — when sending a reply, set the outbound `headers['In-Reply-To']` and `headers['References']` so the reply threads correctly in clients like Gmail / Apple Mail / Outlook:

```ts
await mail.send({
  to: mail.from,
  subject: `Re: ${mail.subject}`,
  text: '...',
  headers: {
    'In-Reply-To': mail.messageId ? `<${mail.messageId}>` : '',
    References: [...mail.references, mail.messageId].filter(Boolean)
      .map(id => `<${id}>`).join(' '),
  },
})
```

(Alibaba DirectMail drops custom headers — the threading approach only works on Resend / SendGrid / Mailgun.)

## Attachments

`attachments` is always a populated array (empty when there are none). Each entry has:

- `filename` — provider-supplied; sanitise before persisting (`path.basename`, allowlist extensions).
- `contentType` — provider-supplied MIME type.
- `content` — Node `Buffer` of the actual bytes. Postmark delivers base64 in the JSON body; the parser decodes it. Mailgun delivers binary multipart parts; the parser reads them.
- `size` — `content.length`.
- `cid` — Content-ID for inline images (angle brackets stripped), absent on regular attachments.

Don't trust `filename` or `contentType` for security decisions — they're attacker-controlled. Verify the actual bytes (`file-type` for magic-number sniffing, an antivirus pass for anything you'll re-distribute).

## Storage suggestion

Two patterns work well in practice:

1. **Decode → process → forget.** Treat the parsed mail as a transient event; persist whatever downstream entity you're creating (a support ticket, a comment, a forum reply) but not the raw webhook body. Cheaper, simpler, no PII-retention question.
2. **Persist the raw body first, parse later.** For workflows where you might re-process (e.g. a parser bug fix) or audit, store the raw body + headers in object storage keyed by `providerMessageId`, then parse and process. Costs storage but makes incident response easy.

Either way, dedupe on `providerMessageId` (or `messageId` if the provider doesn't expose its own) — Mailgun's retry on 5xx will deliver the same message multiple times, and you don't want N support tickets per inbound mail.
